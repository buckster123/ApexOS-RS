## Security

- ✅ **DONE — `read_file`/`list_dir` path confinement** — `tools.rs::confine()` is now the single FS-tool gate: reads confined to workspace + a small allowlist (EDK parts, `/sys`, `/proc/cpuinfo`) minus a secret denylist (`/proc/*/environ`, `/etc/agentd/env`, `~/.ssh`, `/etc/shadow`, `*.api_key`); writes/deletes are workspace-only. `/proc/self/environ` token exfil is dead. **[was high]**
- ✅ **DONE — `http_fetch` SSRF protection** — already had `ssrf_guard` (resolves host, blocks loopback/link-local/RFC1918) + a streaming 4MB cap (not post-buffer). This PR adds a redirect policy that re-runs the guard on each hop. Residual: DNS-rebind TOCTOU (needs a pinned-IP connector). **[was high → low residual]**
- ✅ **DONE — Command injection via `bootstrap_node`** — partially mis-scoped in the original finding: `ssh_password`/`repo_url`/`api_key` were already `shell_single_quote`'d (the `{:?}` was gone). The **live** hole was **`ssh_user`** — interpolated **raw** into the root-run remote scripts (`/home/<user>/ApexOS` in the `sudo -S bash -c …` prep/install commands) *and* the ssh destination, so e.g. `ssh_user="x; rm -rf / #"` was a remote root RCE + argv option-injection. Fix: validate `target_ip` (`is_valid_host`) and `ssh_user` (`is_valid_username`) — reject shell metacharacters / leading `-` rather than escape (never legitimate in a host/user) — and add `--` before the ssh destination (defence-in-depth vs option-injection). Tests: `bootstrap_node_rejects_injection_in_user_and_host`. TOFU (`StrictHostKeyChecking=accept-new`) **retained on purpose** — strict known_hosts would defeat fresh-node provisioning over LAN; the bind is LAN-local + token-gated. The sibling vast.ai tunnel SSH (`:1311`) is argv-only (no shell) with API-sourced host/port → not the same vector. **[was high]**
